## 3. The Full Flow: Plane Issue → Sprint-Ready → Done

This is the flow the PM orchestrates. It's the core of what "work
coordination" means.

### Phase 1: Work Arrives

Work can arrive from multiple sources — the PO works however they want:
- PO creates issue in Plane → sync detects → PM heartbeat sees it
- PO creates task directly on ops board → PM heartbeat sees it
- PO posts directive in board memory → PM heartbeat sees it
- PO tells PM verbally (via chat/comment) → PM creates tasks
- Agent discovers work (subtask, follow-up) → PM triages

The PM doesn't prescribe HOW work arrives. The PM RESPONDS to work
wherever it appears.

### Phase 2: PM Evaluates and Sets Up (readiness 0-20, conversation)

For each new piece of work, PM evaluates:
- What is being asked? (read description, verbatim if available)
- Is it clear enough to work on? Or vague?
- What type? (epic/story/task/bug/spike)
- How complex? (story points estimate)
- Who should work on it? (match capability to content)

PM creates ops board task (if not already there) and sets ALL fields:
- task_type
- task_stage (based on clarity: clear → reasoning, needs analysis → analysis, vague → conversation)
- task_readiness (initial estimate based on clarity)
- story_points
- agent_name (matched by capability)
- requirement_verbatim (PO's exact words)
- delivery_phase (inherited or assessed)
- parent_task (if subtask)
- dependencies (if blocked by other work)

If vague or unclear → stage=conversation, readiness=5-10. Agent asks
PO questions. PM monitors and routes answers.

If epic → PM breaks down into subtasks via fleet_task_create, each
with proper fields and dependencies.

### Phase 3: Pre-Work Builds Readiness (readiness 20-80)

This is the key insight: work at readiness 0-80 is NOT idle. It's
actively being PREPARED through methodology stages.

**Analysis stage (readiness 20-50):**
- Assigned agent examines the codebase, existing implementation
- Produces analysis artifacts (findings, implications, gaps)
- PM monitors: is analysis progressing? Are there blockers?
- When analysis complete → PM can advance readiness

**Investigation stage (readiness 50-80):**
- Agent researches options, explores approaches
- Produces investigation artifacts (options, tradeoffs)
- PM routes 50% checkpoint to PO (informational notification)
- PM monitors: are multiple options explored? PO input received?

These are REAL tasks on the ops board. Not placeholders. Agents do
real work during analysis and investigation — they just don't produce
code. They produce documents, findings, research that INFORM the
eventual implementation.

### Phase 4: Sprint Ready (readiness 80-99, reasoning)

At readiness 80+, the task enters reasoning stage:

**Contribution orchestration:**
- Brain creates contribution tasks for specialists:
  - Architect: design_input (approach, file structure, patterns)
  - QA: qa_test_definition (test criteria the implementation must satisfy)
  - UX: ux_spec (if user-facing work)
  - DevSecOps: security_requirement (if security-relevant)
- Specialists contribute IN PARALLEL
- PM monitors contribution completeness
- PM creates missing contribution tasks if brain didn't auto-create

**Plan production:**
- Agent produces implementation plan referencing verbatim requirement
- Plan specifies target files, steps, acceptance criteria mapping
- PM reviews: does the plan address the requirement?

**PO gate (readiness 90):**
- PM routes gate to PO: "Task X is ready for work. Plan: {summary}.
  Contributions received: architect ✓, QA ✓, DevSecOps ✓.
  Recommend approval."
- PO reviews and approves → readiness advances to 99
- PO rejects → readiness regresses, PM coordinates rework

**Sprint inclusion:**
- Tasks at readiness >80% with contributions complete are "sprint ready"
- PM includes them in current or next sprint
- This is where sprint planning and velocity tracking matter

### Phase 5: Work (readiness 99-100)

PO approved, contributions received, plan confirmed:
- Task dispatched to agent with full autocomplete chain
- Agent implements following the plan
- Agent satisfies QA test criteria, follows architect design,
  follows DevSecOps requirements
- fleet_commit per logical change → fleet_task_complete when done

### Phase 6: Review → Done

- fleet_task_complete triggers full chain (push, PR, approval, etc.)
- Fleet-ops reviews against verbatim, trail, phase standards
- QA validates predefined tests
- DevSecOps security review
- Fleet-ops approves → done
- If rejected → readiness regressed, agent reworks

## 4. What "Fully Autonomous" Mode Means for PM

In fully autonomous mode (work_mode=full-autonomous):

### PM scans Plane proactively
- New Plane issues not yet on ops board → PM evaluates and creates ops board tasks
- Plane priority changes → PM adjusts ops board priorities
- Plane sprint/cycle data → PM incorporates into sprint planning
- Module progress → PM tracks and reports

### PM creates pre-work tasks
- A Plane epic at readiness 0 doesn't need to wait for work stage
- PM creates ANALYSIS tasks on ops board for the epic's scope areas
- PM creates INVESTIGATION tasks for research needed
- These pre-work tasks have their own readiness progression
- They feed into the epic's readiness when completed

### PM manages across both surfaces
- Plane = planning surface (issues, sprints, modules, priorities)
- Ops board = execution surface (tasks with agents, methodology, reviews)
- PM keeps them in sync and makes decisions about what moves where when

### When Plane is NOT connected
- PM still does everything on the ops board alone
- Sprint management via board memory and task tracking
- No Plane scanning, but all other PM functions work
- Plane is an enhancement, not a requirement

## PO Answers to Design Questions (2026-04-07)

### 1. Sprint ready threshold
**Answer: Yes.** Add explicit `sprint_ready_threshold: 80` to methodology.yaml.
The reasoning stage range [80, 99) implies sprint readiness, but making it
explicit in config is better — configurable, clear, PO can adjust.

### 2. PM intelligence with Plane (NOT brainless auto-creation)
**Answer: Much more complex.** The PM is a top-tier Scrum Master. Not a
mechanical queue processor. A good PM is INTELLIGENT and COMMUNICATIVE:

- Anticipates what the PO will need for upcoming sprints
- Prepares backlogs proactively — grooms, refines, re-estimates, re-prioritizes
- Communicates with PO about what to prepare next
- Thinks about sprint goals, capacity, dependencies
- Balances urgent vs important
- Plans ahead, doesn't just react

The PM doesn't auto-create a task for every Plane issue. They EVALUATE:
- Is this relevant for the current sprint or future sprints?
- Does the PO need to weigh in before this moves forward?
- What level of preparation does this need?
- Who should be involved in evaluating it?
- What's the right entry point (conversation? analysis? reasoning?)

Fleet-ops is at the OTHER end — the "back end" of monitoring, health,
infra status, quality. PM is the "front end" — planning, coordination,
PO communication, sprint preparation.

### 3. Subtask vs independent task — work hierarchy and artifact ownership
**Answer: It depends. Per-case evaluation.** The key concept is LEVELS:

```
Root (epic / module)
├── Branch (story / issue)
│   ├── Leaf (subtask)
│   └── Leaf (subtask)
└── Branch (story / issue)
    └── Leaf (subtask)
```

**Rules for the PM's evaluation:**

- **Does it need its own artifact?** → Subtask of the parent. Keeps
  artifacts from colliding. Each agent works on their own subtask with
  their own artifact.

- **Is it just responding to a comment?** → Not necessarily a task.
  Unless the comment itself leads to the NEED for a task. A comment
  saying "looks good" doesn't need a task. A comment saying "this
  approach won't work because X" might need an investigation task.

- **At which LEVEL are you working?** The PM must know:
  - Root level: the big picture (epic, module)
  - Branch level: a specific piece of work (story, issue)
  - Leaf level: a specific contribution (subtask, analysis, test)

- **Artifact collision prevention:** If two agents work at the same
  level on the same task, their artifacts conflict. Proper breakdown
  means: each task = one agent = one artifact owner. The PM ensures
  this by breaking down to the right granularity.

- **Is it an epic, a module, an issue?** Same evaluation process —
  what level, what needs its own artifact, who works at which branch.

This is an EVALUATION PROCESS, not a mechanical rule. The PM's Scrum
Master intelligence determines the right breakdown per situation.

## 11. Deeper Analysis: PM as Intelligent Scrum Master

The PM is NOT a task dispatcher. The PM is a strategic coordinator who
THINKS about the work. This changes how PM files should be written.

### What a top-tier PM ACTUALLY does

**Backlog grooming (proactive, not reactive):**
- Reviews upcoming work before it's urgent
- Estimates effort on tasks before they're assigned
- Identifies dependencies between tasks BEFORE they block
- Suggests task ordering based on risk and value
- Prepares the PO for decisions that will be needed

**Sprint planning intelligence:**
- Assesses capacity: which agents are available, which are loaded
- Balances the sprint: not all hard tasks, not all easy tasks
- Plans for contributions: if a story needs architect + QA + DevSecOps,
  PM ensures those agents have capacity for contribution tasks
- Considers velocity: historical data informs sprint sizing
- Accounts for unknowns: leaves capacity for emergent work

**PO communication (strategic filtering):**
- Doesn't dump raw data on PO
- Summarizes: "3 tasks are sprint-ready, 2 need your input on scope,
  1 is blocked on a decision only you can make"
- Prepares context: when routing a gate, PM provides the summary,
  the plan, the contribution status, the recommendation
- Follows up: if PO hasn't responded to a gate, PM reminds with
  clear urgency indication

**Inter-agent orchestration (not just assignment):**
- Knows that architect needs to contribute before engineer implements
- Knows that QA needs to predefine tests before work starts
- Actively monitors: "architect hasn't contributed design input for
  Task X yet — creating a contribution task" or "checking with architect"
- Resolves conflicts: if architect says one thing and DevSecOps says
  another, PM facilitates resolution (or escalates to PO)

**Work hierarchy intelligence:**
- Evaluates every piece of work: what level? what breakdown needed?
- Epics → stories → tasks → subtasks — proper decomposition
- Each level has clear artifact ownership
- Dependencies flow correctly (design before implement, test before merge)
- No two agents collide on the same artifact

### What this means for PM CLAUDE.md

The PM CLAUDE.md shouldn't describe a mechanical process
("for each unassigned task: set fields, assign"). It should describe
the PM's INTELLIGENCE:

- How to evaluate work at different levels
- How to decide the right entry point and breakdown
- How to communicate strategically with the PO
- How to orchestrate contributions proactively
- How to manage sprints with capacity and velocity awareness
- When to create tasks vs when to just respond to a comment

This is harder to write than a mechanical process. But it's what
makes the difference between a generic dispatcher and a top-tier PM.

## 12. Deeper Analysis: Fleet-Ops as Ops Board Lead

Fleet-ops is at the OTHER end from PM:

**PM (front end):** planning, coordination, PO communication, sprint prep
**Fleet-ops (back end):** quality, reviews, monitoring, health, compliance

### What fleet-ops ACTUALLY does as ops board lead

**Quality guardian:**
- REAL reviews — reads actual work, compares to verbatim requirement
- Trail verification — did the task follow all required stages?
- Contribution verification — were all required inputs received?
- Phase standard verification — does the work meet delivery phase quality?

**Methodology compliance:**
- Monitors that agents follow stage protocols
- Flags violations (code during analysis, skipped stages)
- Reports patterns to board memory for immune system

**Board health:**
- Tasks stuck too long → action (flag PM, escalate)
- Agents offline with work → alert
- Review queue management → process promptly
- Budget awareness → alert when spending is concerning

**Infrastructure monitoring:**
- Fleet health: orchestrator, gateway, daemons
- Service health: MC, Plane, IRC, LocalAI
- Security health: in coordination with DevSecOps

This is a completely different role from PM. They overlap on the ops
board but with distinct authorities and responsibilities.
