# Phase A: Code Delivery Pipeline

## Problem

Agent commits to local worktree → nothing happens. No push, no PR, no visibility.
The NNRT fix (8223d7c) exists in a local branch that nobody can see.

## Current Flow (Broken)

```
Task created → dispatched → agent works in worktree → commits locally → posts MC comment → "review"
                                                                                            ↓
                                                                                         (dead end)
```

## Target Flow

```
Task created → dispatched → agent works in worktree → commits → pushes branch → creates PR
                                                                                    ↓
                                                                              PR URL in MC task
                                                                                    ↓
                                                                              Channel notification
                                                                                    ↓
                                                                              Human reviews PR
                                                                                    ↓
                                                                              Merge → task done
```

## Design Decisions

### Where does push/PR happen?

**Option A: Agent does it (in SOUL.md/workflow)**
- Agent runs `git push` and `gh pr create` as part of task completion
- Pro: Self-contained, no external automation
- Con: Agent needs git push access + gh auth. More token management.
  Agent might mess up the PR description. More token exposure.

**Option B: Post-completion hook script**
- When task moves to "review", a script auto-pushes and creates PR
- Could be triggered by monitoring MC activity (polling or SSE)
- Pro: Centralized control, consistent PR format, single auth point
- Con: Another daemon to run. Polling latency.

**Option C: integrate-task.sh becomes the standard flow**
- Agent completes → human or automation runs integrate-task.sh
- Pro: Human in the loop for push/PR
- Con: Manual step, defeats the purpose of automation

**Recommended: Option A (agent does it) with guardrails**

The agent already has `gh` access (it's in the workspace). The fleet-commit skill
already teaches commit format. We add push+PR to the agent's completion workflow.
Guardrails:
- Agent can only push to `fleet/*` branches (never main)
- PR always targets main/default branch
- PR description follows a template
- Push credentials: use the NNRT repo's existing git auth (ssh key or gh auth)

### Git Authentication

For agents to push, they need git write access. Options:
- SSH key in the workspace (security concern)
- `gh auth` CLI (already works if `gh` is installed and authed)
- Git credential helper (from system config)

The simplest: ensure `gh auth status` works on the machine. If the human has
`gh` authed, agents inherit that. No extra token management.

### PR Description Template

Every fleet PR should have:
```markdown
## Fleet Task: {task_title}

**Task ID:** `{task_id}`
**Agent:** {agent_name}
**Board:** Fleet Operations
**MC URL:** http://localhost:3000/boards/{board_id}/tasks/{task_id}

### Changes
{git diff --stat output}

### Summary
{agent's completion summary from MC comment}

---
Generated by [OpenClaw Fleet](https://github.com/cyberpunk042/openclaw-fleet)
```

### MC Task Custom Fields

After PR is created, update the MC task with:
- `pr_url` — the GitHub PR URL
- `branch` — the branch name
- `project` — the project name

This requires MC custom fields to be set up on the board.

## Milestones

### M61: Agent Push + PR on Completion

**Tasks:**
1. Update MC_WORKFLOW.md: after marking task "review", agent must:
   - `git push -u origin fleet/<agent>/<task-short>`
   - `gh pr create --title "fleet(<agent>): <task-title>" --body "<template>"`
   - Post PR URL as task comment
2. Update fleet-task-update skill: add push+PR step to completion workflow
3. Verify `gh auth status` works on the host (add to pre-flight checks in setup.sh)
4. Test: dispatch task → agent completes → branch pushed → PR created

### M62: MC Custom Fields for Fleet Tasks

**Tasks:**
1. Study OCMC custom fields API — how to create, populate, display
2. Create fleet custom fields on the board: project, branch, pr_url
3. Update create-task.sh to set project field on creation
4. Update agent workflow to set branch and pr_url on completion
5. Script board configuration in setup-mc.sh

### M63: Post-PR Notification

**Tasks:**
1. Agent posts PR URL to board memory with tags: ["pr", "review", "<project>"]
2. If IRC/Discord configured: agent posts to channel with PR link
3. MC activity shows PR creation event
4. Human can see all pending PRs from board memory or channel

### M64: PR Template and Quality

**Tasks:**
1. Create `.github/PULL_REQUEST_TEMPLATE/fleet.md` in target projects
2. Agent uses template when creating PR
3. PR includes: task link, agent name, diff stat, summary
4. Consider: automated checks on PR (GitHub Actions trigger)
