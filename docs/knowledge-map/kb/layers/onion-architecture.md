# Onion Architecture — 8-Layer Agent Context Injection

**Type:** Architecture Pattern
**System:** S22 (Agent Intelligence), S20 (Infrastructure — gateway)
**Source:** fleet-elevation/02-agent-architecture.md

## Purpose

The 8-layer onion defines the ORDER in which content is injected into an agent's system prompt by the gateway. Files injected EARLY have more influence on AI behavior (Lost in the Middle effect). This ordering creates a funnel from identity → values → rules → capabilities → team → state → task → action, narrowing the AI's response space so the correct action is the most natural continuation.

## The 8 Layers

```
Layer 1: IDENTITY.md    → WHO I AM (grounding — most influence)
Layer 2: SOUL.md        → WHAT I VALUE (boundaries)
Layer 3: CLAUDE.md      → WHAT I MUST DO (role rules, max 4000 chars)
Layer 4: TOOLS.md       → WHAT I CAN DO (chain-aware capabilities)
Layer 5: AGENTS.md      → WHO I WORK WITH (synergy knowledge)
Layer 6: fleet-context   → WHAT'S HAPPENING (dynamic fleet state)
Layer 7: task-context    → WHAT I'M WORKING ON (autocomplete chain)
Layer 8: HEARTBEAT.md   → WHAT TO DO NOW (action prompt — last)
```

## Layer Details

### Layer 1: IDENTITY.md (Inner — constant)
- Injected FIRST — grounds all subsequent output
- Contains: name, fleet_id, role, specialty, personality, place in fleet
- "You are Software Engineer, Fleet Alpha. Top-tier expert."
- NEVER changes during operation
- Standard: identity-soul-standard.md (4 required sections)

### Layer 2: SOUL.md (Inner — constant)
- Values and behavioral boundaries
- 10 anti-corruption rules (exact text from fleet-elevation/20)
- Per-role values + humility clause
- NEVER changes during operation
- Standard: identity-soul-standard.md (5 required sections)

### Layer 3: CLAUDE.md (Middle — stable)
- Role-specific rules (max 4000 chars — gateway enforced)
- 8 sections: core responsibility, role rules, stage protocol, tool chains, contribution model, boundaries, context awareness, anti-corruption summary
- Each agent's CLAUDE.md is UNIQUE to their role
- Changes when fleet evolves, not during operation
- Standard: claude-md-standard.md

### Layer 4: TOOLS.md (Middle — stable)
- Chain-aware tool reference (NOT a list — documents what HAPPENS when each tool is called)
- Per tool: what it does, what chain fires, when to use, what inputs, what you do NOT need to call manually
- GENERATED by scripts/generate-tools-md.sh from tools.py + fleet-elevation/24
- Different per role (architect uses different tools than engineer)
- Standard: tools-agents-standard.md

### Layer 5: AGENTS.md (Middle — stable)
- Knowledge of 9 colleagues
- Per colleague: contributes to me (what + when), I contribute to them (what + when), when to @mention
- GENERATED by scripts/generate-agents-md.sh from synergy matrix
- Bidirectional consistency enforced
- Standard: tools-agents-standard.md

### Layer 6: fleet-context.md (Outer — dynamic, every 30s)
- GENERATED by orchestrator (preembed.py → context_writer.py)
- Contains: fleet state (work_mode, cycle_phase, backend_mode, budget_mode), PO directives, @mentions, agent's assigned tasks (FULL detail), role-specific data, events since last heartbeat
- NEVER compressed — full data always (PO requirement)
- Standard: context-files-standard.md (7 required sections)

### Layer 7: task-context.md (Outer — dynamic, at dispatch)
- GENERATED by orchestrator (context_assembly.py → context_writer.py)
- The AUTOCOMPLETE CHAIN: identity grounding → verbatim requirement → stage protocol → contributions → confirmed plan → artifact state → action directive
- ORDER IS CRITICAL — drives correct behavior through natural continuation
- Standard: context-files-standard.md (10 sections in fixed order)

### Layer 8: HEARTBEAT.md (Middle — stable, LAST)
- Action protocol — what to do NOW after absorbing all context
- Priority order: PO directives → messages → core job → proactive contributions → health → HEARTBEAT_OK
- Injected LAST so it drives the immediate next action
- 5 types: PM, fleet-ops, architect, devsecops, worker (6 variants)
- Standard: heartbeat-md-standard.md

## Onion Layers (dependency direction)

```
┌─────────────────────────────────────────┐
│           OUTER: Dynamic Data            │
│   fleet-context.md, task-context.md      │
│   Refreshed every 30s. References inner. │
├─────────────────────────────────────────┤
│         MIDDLE: Rules & Patterns         │
│   CLAUDE.md, HEARTBEAT.md, TOOLS.md,     │
│   AGENTS.md. Stable. References inner.   │
├─────────────────────────────────────────┤
│         INNER: Identity (constant)       │
│   IDENTITY.md, SOUL.md, agent.yaml       │
│   Never changes. References nothing.     │
└─────────────────────────────────────────┘

Dependencies point INWARD:
  Outer references middle and inner
  Middle references inner
  Inner references nothing — IS the foundation
```

## Relationships

- IMPLEMENTED BY: gateway executor.py + ws_server.py (_build_agent_context)
- GENERATED BY: scripts/provision-agent-files.sh (static files), orchestrator (dynamic context/)
- READ BY: gateway at agent heartbeat/dispatch (injects into system prompt)
- DRIVES: autocomplete chain engineering (§36)
- PREVENTS: Lost in the Middle (rules at start succeed, buried rules fail)
- CONNECTS TO: knowledge map injection profiles (how MUCH of each layer to inject per context size)
- CONNECTS TO: SessionStart hook (can augment layer content dynamically)
- CONNECTS TO: InstructionsLoaded hook (can modify CLAUDE.md content at load time)
- VALIDATES: scripts/validate-agents.sh (all files exist, no concern mixing, char limits)

## Anti-Corruption Through Structure

Each layer prevents specific diseases:
- IDENTITY (Layer 1): prevents identity confusion, role drift
- SOUL (Layer 2): prevents value violation, anti-corruption bypass
- CLAUDE.md (Layer 3): prevents methodology bypass, scope creep
- TOOLS.md (Layer 4): prevents wrong tool usage, missed chains
- AGENTS.md (Layer 5): prevents contribution ignoring, siloed work
- fleet-context (Layer 6): prevents stale state, missed directives
- task-context (Layer 7): prevents verbatim deviation, plan drift
- HEARTBEAT (Layer 8): prevents priority inversion, HEARTBEAT_OK before checking
