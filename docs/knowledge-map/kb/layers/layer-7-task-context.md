# Layer 7: task-context.md

**Type:** Context File (Outer layer — dynamic, written at dispatch)
**Position:** 7 of 8 (the autocomplete chain — drives correct work behavior)
**System:** S07 (Orchestrator), S19 (Session), S22 (Agent Intelligence)
**Standard:** context-files-standard.md
**Location:** agents/{name}/context/task-context.md

## Purpose

The AUTOCOMPLETE CHAIN. Content arranged so the AI's most natural continuation IS the correct professional action. This is the core design principle — structure prevents deviation more reliably than rules. The ORDER of sections is FIXED and CRITICAL.

## Required Sections (10, ORDER IS CRITICAL)

```
1.  YOU ARE: {name} ({role})              ← identity grounding
2.  YOUR TASK: {title}                     ← task focus
3.  YOUR STAGE: {stage}                    ← stage awareness
4.  READINESS: {readiness}%               ← authorization level
5.  VERBATIM REQUIREMENT                   ← PO's exact words (SACROSANCT)
    "{requirement_verbatim}"
    (Do not interpret. Do not rephrase.)
6.  STAGE PROTOCOL                         ← MUST/MUST NOT/CAN from stage_context.py
7.  INPUTS FROM COLLEAGUES                 ← contributions received
    Design Input (from architect): ...
    Test Criteria (from QA): ...
    Security Requirements (from DevSecOps): ...
8.  DELIVERY PHASE: {phase}                ← PO-declared maturity level
9.  WHAT TO DO NOW                         ← action directive
10. WHAT HAPPENS WHEN YOU ACT              ← chain documentation
```

**Why this order works:** AI reads 1-10 and the MOST NATURAL next token is: "Let me start with step 1 of my confirmed plan, referencing the architect's design input and QA's test criteria..." Deviation requires FIGHTING the context structure.

## Generation Pipeline

```
Orchestrator dispatches task:
├── context_assembly.assemble_task_context(task, mc, board_id)
│   ├── task core: id, title, status, priority, description
│   ├── custom fields: readiness, stage, verbatim, phase, agent
│   ├── methodology: stage instructions from stage_context.py
│   ├── artifact: current state + completeness from Plane HTML
│   ├── comments: typed, contributor attribution
│   ├── related tasks: parent, children, dependencies
│   └── Plane data: sprint, issue details
├── [future] map_navigator.py: role+stage → knowledge map content
├── preembed.py: format_task_preembed()
└── context_writer.py: write_task_context()
```

## Relationships

- GENERATED BY: orchestrator dispatch, context_assembly.py, preembed.py, context_writer.py
- INJECTED BY: gateway (_build_agent_context, position 7, max 8000 chars)
- READ BY: agent when working on dispatched task
- DATA FROM: mc_client.py (task, comments), plane_client.py (artifact HTML), methodology.py (stage), stage_context.py (instructions), contributions.py (received contributions)
- FEEDS: agent's work behavior (the autocomplete chain drives action)
- CONNECTS TO: autocomplete.py (assembles this content in correct order)
- CONNECTS TO: knowledge map navigator (future — map-driven content sections)
- CONNECTS TO: fleet_contribute (contributions appear in section 7)
- CONNECTS TO: standards.py (artifact type standards for section 8)
- CONNECTS TO: methodology.py (stage protocol for section 6)
- DOES NOT CONTAIN: fleet state (Layer 6), tool docs (Layer 4), action protocol (Layer 8)
- DYNAMIC: written at dispatch, refreshed when task context changes
- THE VERBATIM IS SACROSANCT: requirement_verbatim is NEVER modified, summarized, or interpreted

## Anti-Corruption Through Chain Order

| Disease | How Chain Prevents It |
|---------|----------------------|
| Deviation | Verbatim appears at position 5 — anchors ALL subsequent work |
| Laziness | Acceptance criteria listed explicitly at position 5 |
| Compression | Full pre-embed — never summarized (PO requirement) |
| Scope creep | Plan is explicit (section 9), verbatim defines exact scope |
| Abstraction | Verbatim is LITERAL — exact words |
| Not listening | Contributions pre-embedded at position 7 — prominent, not optional |
