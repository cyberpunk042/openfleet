# Layer 5: AGENTS.md

**Type:** Agent File (Middle layer — stable, GENERATED)
**Position:** 5 of 8 (team knowledge — agent knows colleagues before seeing state)
**System:** S22 (Agent Intelligence)
**Standard:** tools-agents-standard.md

## Purpose

Knowledge of 9 colleagues and synergy points. Each colleague listed with: what they contribute TO you (and when), what you contribute TO them (and when), when to @mention them. Bidirectional consistency enforced — if architect says "I contribute design_input to engineer," engineer says "I receive design_input from architect."

## Generation

GENERATED by `scripts/generate-agents-md.sh` from fleet-elevation/15 (synergy matrix) + config/agent-identities.yaml (display names). Never hand-written.

## Required Format Per Colleague (3 fields)

```
### {Display Name} ({role})
**Contributes to me:** {what} — {when}
**I contribute to them:** {what} — {when}
**When to @mention:** {specific conditions}
```

## Relationships

- INJECTED BY: gateway (_build_agent_context, position 5)
- GENERATED BY: scripts/generate-agents-md.sh
- SOURCE: fleet-elevation/15 (cross-agent synergy), config/agent-identities.yaml
- CONFIGURED BY: config/synergy-matrix.yaml (who contributes what to whom)
- VALIDATED BY: scripts/validate-agents.sh (9 colleagues listed, bidirectional consistency, display names match agent.yaml)
- FEEDS: agent's contribution awareness (knows who to @mention, what to expect)
- CONNECTS TO: fleet_contribute tool (contribution types match AGENTS.md)
- CONNECTS TO: fleet_request_input tool (knows which role to request from)
- CONNECTS TO: CLAUDE.md Layer 3 (contribution model section, briefer)
- CONNECTS TO: orchestrator Step 4b (brain creates contribution subtasks per this matrix)
- DOES NOT CONTAIN: identity, values, rules, tool docs, dynamic data, action protocol
- STABLE: regenerated when synergy matrix changes
- PER AGENT: 10 unique AGENTS.md files (different synergy per role)
