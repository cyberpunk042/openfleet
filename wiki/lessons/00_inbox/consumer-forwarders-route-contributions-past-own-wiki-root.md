# Consumer Forwarders Must Route Contributions Past Their Own Wiki Root

## Summary

When a consumer project generates a tool forwarder to a central hub's CLI (e.g., `tools/gateway.py` in OpenFleet forwarding to the research wiki's `gateway.py`), the forwarder typically auto-adds `--wiki-root /path/to/consumer` so the hub tool operates on the consumer's local data. That's correct for most commands — lint, validate, status, navigate. It's WRONG for commands whose natural target is the hub itself — most notably `contribute`. A generic forwarder that treats all commands the same silently misroutes contributions into the consumer's own wiki, where they sit invisible to the hub's review pipeline.

## Context

Any ecosystem where:
- A central hub provides a CLI tool that operates in two scopes: "this local project" and "the hub itself"
- Consumer projects install a forwarder (generated by the hub or hand-written) that wraps the hub's CLI
- The forwarder auto-adds the consumer's local path so operations default to the consumer's scope
- At least one command exists whose natural target is the HUB, not the consumer

Examples of hub-targeting commands in an LLM Wiki context:
- `contribute` (lessons / remarks / corrections go to the hub's intake)
- `feedback` / `upvote` / `endorse` (operator-style signals TO the hub)
- Anything that creates a pending-review artifact in the hub's maturity pipeline

## Insight

> [!warning] Generic auto-routing breaks for hub-bound commands
>
> The forwarder pattern "always add `--wiki-root $PWD`" is a good default for queries and audits (operate on local data). For contributions, it inverts the semantics: a contribution is meant FOR THE HUB, filed by a LOCAL SOURCE. Auto-passing `--wiki-root local` makes the hub tool write TO THE LOCAL wiki instead of FROM the local source TO the hub.
>
> The fix has two sides:
>
> 1. **Hub-side default**: the hub's `contribute` command defaults target to `brain`/`hub`, not to the `--wiki-root` it was passed. Consumer's `--wiki-root` is used only to stamp `contribution_source` metadata (who filed it), not to choose where it lands. Consumers opt into local contribution via `--target local`. This is what the brain landed in commit `154bc58`.
>
> 2. **Consumer-side fallback**: if the hub hasn't been updated yet, invoke the hub's CLI DIRECTLY (bypass the forwarder) by cd-ing to the hub directory and running its venv python — no `--wiki-root` passed.

## Evidence

> [!bug]- OpenFleet contributions misfiled 2026-04-17
>
> **What happened:** I (solo session on OpenFleet, 2026-04-17) ran `python3 -m tools.gateway contribute --type correction --title ...` and `--type remark --title ...`. Expected: contributions land in brain's `wiki/log/`. Actual: contributions landed in `openfleet/wiki/log/`. Both misroutes went unnoticed until I inspected filesystem paths after filing.
>
> **Verification:** `ls /home/jfortin/devops-solutions-research-wiki/wiki/log/` showed no new entries; `ls /home/jfortin/openfleet/wiki/log/` showed the two new files.
>
> **Root cause:** OpenFleet's `tools/gateway.py` is a forwarder generated by the brain's `setup.py --connect`. It unconditionally adds `--wiki-root /home/jfortin/openfleet` to every invocation. Brain's `op_contribute` (at the time I first ran the forwarder) resolved `paths["wiki"]` from `--wiki-root`, so it wrote to OpenFleet's wiki instead of brain's.
>
> **Fix applied (consumer side):** Invoked brain's CLI directly: `cd /home/jfortin/devops-solutions-research-wiki && .venv/bin/python -m tools.gateway contribute ...` (no `--wiki-root`). Both contributions re-filed. Misfiled copies removed from `openfleet/wiki/log/`.
>
> **Fix applied (hub side):** Per PO clarification 2026-04-17, the brain had already landed the fix in commit `154bc58`: `op_contribute` signature now takes `target: str = "brain"`, `wiki_dir = paths.get("brain_wiki") or paths["wiki"]` when target is `brain`. `--target local` still works for self-contribution.
>
> **File trail of evidence:** `openfleet/wiki/log/walkthrough-c-(openfleet)-in-goldilocks-flow:-ground-truth-v.md` (deleted), `openfleet/wiki/log/gateway-status-identity-parser:-fragile-string-match-misses-.md` (deleted), brain's `wiki/log/walkthrough-c-(openfleet)-ground-truth-verification-2026-04-.md`, brain's `wiki/log/gateway-identity-parser-fragility-+-forwarder-contribute-tar.md`.

> [!tip]- Why this matters for the feedback loop
>
> A feedback loop where contributions silently land in the wrong place is a loop that does not close. The consumer BELIEVES they contributed; the hub NEVER SEES the contribution; learnings that should accelerate the ecosystem stay trapped at the consumer. Per brain's `Infrastructure > Instructions` principle, an instruction "run contribute with the right args" achieves ~25% correctness in the face of opaque routing; an infrastructure default that routes correctly by default achieves 100%.

## Applicability

| Context | Apply this lesson |
|---------|------------------|
| LLM Wiki ecosystems with consumer forwarders | Audit every forwarder: which commands target LOCAL, which target HUB. Make the hub-bound commands route-by-default to the hub regardless of `--wiki-root`. |
| MCP servers that expose local-scope and hub-scope tools | Same pattern. Hub-bound MCP tools should resolve the hub path independently of any consumer-passed local context. |
| Multi-project CI that forwards commands to a central orchestrator | Distinguish `ci-lint` (operates on the calling project) from `ci-publish-result` (writes back to orchestrator). The latter must not use the former's path defaults. |
| Any wrapper script that auto-adds a scope flag | Name the commands that should bypass the auto-addition. Default to conservative behavior (bypass when unsure); opt INTO the auto-added scope when safe. |

| When NOT to apply | Why |
|---|---|
| Pure-query forwarders (lint, validate, search) | The `--wiki-root` default IS correct for queries — they operate on local data. No change needed. |
| Single-scope hubs (no local/hub distinction) | If every command operates on one canonical scope, there's nothing to route past. The lesson assumes a two-scope ecosystem. |

> [!info] Self-check before trusting a forwarder for a new command
>
> 1. What scope does this command WRITE to?
> 2. Is that scope LOCAL (the consumer) or HUB (the central repo)?
> 3. Does the forwarder force one scope via `--wiki-root` or similar?
> 4. If forced AND the natural write-target differs, the command is misrouted. Bypass the forwarder or update it.
